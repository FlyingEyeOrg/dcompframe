# 实现原则

## 1. 先测后码

所有新增功能先由单元测试定义行为，再实现最小通过代码。

## 2. 完整能力优先

在保持清晰分层的前提下，优先补齐 development-tasks 中定义的模块能力，避免长期骨架化。

## 3. 资源与状态安全

- 使用值语义与明确状态转换。
- 避免隐藏副作用。
- 对关键动作设置前置条件（如提交门禁、窗口句柄有效性、设备状态）。

## 4. Windows 约束优先

- 默认遵循 `WS_EX_NOREDIRECTIONBITMAP`。
- DPI 计算采用 96 作为基线。
- DirectX 初始化路径遵循 D3D11 + DXGI + DComp 的接口契约。

## 5. 可测试性优先

- 模块接口小而清晰。
- 布局、控件状态、动画推进、诊断统计均保持确定性行为，便于断言。

## 6. 渐进集成原则

- 通过 `Simulated` 后端保证跨环境可测。
- 通过 `DirectX` 后端提供真实图形链路入口，`DirectX12` 预留扩展点。
- 使用统一测试保护显式后端选择路径的稳定性。

## 6.1 命令缓冲原则

- 渲染命令先入队后批量 drain，减少 UI 线程上的同步阻塞。
- 先保证命令顺序与可测试性，再做高阶批处理优化。

## 6.2 线程解耦原则

- 渲染线程生命周期必须显式受控（start/stop）。
- 共享队列访问必须通过互斥保护，禁止无锁写入竞态。

## 7. 压测优先原则

- 对设备丢失/恢复场景提供集成级循环压测，优先保证恢复路径可重复、可回归。
- 压测结果必须进入测试文档和开发进度，避免隐性回归。

## 8. 兼容回退原则

- 开发阶段采用 Fail-Fast，不保留自动回退、静默降级与兜底渲染路径。
- 后端选择必须显式（DirectX/Simulated），框架不进行隐式切换。
- 在 `WS_EX_NOREDIRECTIONBITMAP` 场景必须使用正确 DComp 接口探测，避免透明窗口误判。
- D2D 初始化失败与运行时绘制失败均直接返回错误并记录诊断，避免不确定行为。
- 交互可见性原则：hover/press/toggle 等状态必须由窗口消息触发重绘并落到渲染层。

## 9. 多窗口一致性原则

- 任意单窗口销毁不得影响其它窗口消息循环。
- 仅当最后一个窗口销毁时才允许投递全局退出消息。

## 10. 启动稳定性原则

- 在 demo/调试入口中，若首选图形后端初始化失败，直接返回错误并输出诊断，避免掩盖真实问题。

## 11. Panel 容器原则

- `Panel` 提供基础 `arrange` 语义：容器和子元素默认贴合可用区域，用于承载上层自定义布局。

## 12. 表单文本原则

- 单行表单标题、状态值和复选项标签默认禁止自动换行，避免被窄列布局挤压后出现非预期断行。
- 多行说明文本、富文本摘要和卡片正文必须使用单独的 wrap 文本格式，不混用单行输入格式。

## 13. 性能优先滚动原则

- 滚动容器优先裁剪并只绘制可见项，不在每帧全量遍历所有列表内容。
- Popup / dropdown / overlay 使用浮层布局，避免控件展开时触发额外主布局重排。

## 14. 编辑语义一致性原则

- `TextBox` 与 `RichTextBox` 的光标移动、选区替换、退格和删除语义保持一致，避免同类输入控件产生不一致行为。
- 文本编辑结果必须优先由测试锁定，再接入渲染层 caret/selection 可视反馈。

## 15. 响应式演示原则

- Demo 中的交互卡片、滚动区与列表区应基于窗口尺寸动态分配剩余空间，避免固定高度导致内容裁切。
- 中文桌面 UI 默认采用 `Microsoft YaHei`，保证 Element Plus 风格在 Windows 中文环境下的实际可读性。

## 16. Hover 驱动滚动原则

- `ScrollViewer`、`ListView`、`ItemsControl` 的鼠标滚轮滚动必须以 hover 区域为准，禁止依赖历史焦点继续消费滚轮消息。
- 滚动条拖拽与滚轮滚动必须收敛到同一偏移模型，避免两套状态源产生错位。

## 17. 规范对齐控件原则

- `ui-framework-design-spec.md` 中定义的控件能力必须先落状态模型，再落渲染表现，最后落交互测试。
- demo 必须覆盖所有框架内置控件的最小可视与状态展示，避免“有类无演示”。
- 新控件外观统一遵循 Element Plus 视觉体系，同时保持 Win32 + D3D11 + DComp 事件与渲染链路不变。

## 18. 页面级滚动原则

- 当鼠标滚轮未命中内层滚动容器时，应由窗口主内容偏移消费滚轮输入，保证整体页面可继续向下扩展设计。
- 内层滚动优先级高于页面级滚动，避免滚动冲突与穿透。

## 19. 字体一致性原则

- 默认字体族使用 `Microsoft YaHei UI`，字号优先对齐 Element Plus 的 14px 内容体系。
- 输入控件需优先保证文本不裁剪，再进行视觉居中微调。

## 20. 标题与内容分区原则

- demo 页面标题区与内容滚动区必须分离裁剪，页面级滚动不得回卷覆盖标题与主操作区。
- 当交互预览区空间有限时，优先压缩预览区内部留白和说明区高度，不牺牲 `TabControl`、动画示例、滚动容器的最小可见性。

## 21. 滚动条轨道交互原则

- 具有纵向滚动条的控件除 thumb 拖拽外，必须支持轨道点击跳转，避免“可见但不可点”的半成品交互。
- 轨道点击与 thumb 拖拽必须收敛到同一偏移计算模型，保证行为一致和可测试性。

## 22. 布局容器视觉最小化原则

- 布局容器（如 `Panel`）默认不承担视觉装饰，不应附带边框和非透明背景。
- 视觉表现交由具体业务控件承担，避免容器层影响控件视觉统一性。

## 23. 滚动聚焦反馈原则

- 所有带滚动条控件在用户点击轨道或滑块后，必须提供明确聚焦态反馈。
- 聚焦态颜色与普通 hover 状态分层，保证用户能感知当前滚动操控目标。

## 24. 骨架层边界原则

- `Application` 与 `Window` 只提供生命周期与宿主骨架，不直接创建 demo 业务控件。
- 示例窗口的控件树、事件绑定与业务状态必须留在 demo 层实现。

## 25. demo 分区不重叠原则

- demo 页面各 section 的高度分配必须受总可用高度约束，禁止通过多个固定下限叠加导致区块互相侵占。
- 空间不足时优先压缩次要区块，保留关键交互区的最小可见性。

## 26. 鼠标捕获态一致性原则

- 使用 `SetCapture` 开始拖拽后，必须在 `ReleaseCapture` 或收到 `WM_CAPTURECHANGED` 时同步清理视觉捕获态。
- 滚动条、滑块等拖拽控件的高亮色只能表达“当前正在操控”或“当前 hover”，不能在释放后残留。
